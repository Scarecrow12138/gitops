# 发布管理功能执行计划

## 1. 同一发布任务判定

发布任务优先按中台记录标识判断。服务顺序可能变化，失败后重跑时直接用原始顺序做主键容易误判。

当前实现按以下优先级生成 `task_key`：

1. 优先使用中台返回的发布记录 `id` 集合。
2. 如果所有记录都没有 `id`，使用 `relNo` 集合。
3. 如果 `id` 和 `relNo` 都不可用，才使用规范化后的服务内容指纹。

刷新发布任务时：

- `task_key` 命中未完成任务：认为是同一个发布任务，只更新服务和原始记录快照。
- `task_key` 不同：创建新的发布任务。
- 中台返回空：展示暂无待发布任务。

失败后重复发布：

- 不创建新发布任务。
- 在同一个发布任务下新增一条执行记录。

## 2. 数据模型

新增本地表：

- `release_tasks`：保存发布任务、任务标识、服务快照、原始中台记录。
- `release_attempts`：保存每次执行记录、状态、失败步骤、执行日志。
- `release_configs`：保存 Jenkins、提交数阈值等发布参数。

`operatorId = 1800513641113452545` 固定在后端逻辑中，不在发布页展示。

## 3. 后端命令

新增 Tauri 命令：

- `refresh_release_task`：请求中台待发布接口，计算 `task_key`，创建或更新当前发布任务。
- `start_release_attempt`：为当前任务新增执行记录，并按发布流程执行。
- `complete_release_task`：调用中台完成发布接口，并将本地任务标记为成功。

发布流程：

1. 获取并校验合并到 `prep-3.0` 的 MR。
2. 合并源分支到 `prep-3.0`。
3. 创建 `prep-3.0 -> main` MR。
4. 备份 main 分支并合并主干。
5. 触发 `lasen-prod-snapshot-release` 和待发布服务 Jenkins Job。
6. 检查 Jenkins 构建结果。
7. 调用中台完成发布。

## 4. 前端页面

新增菜单「发布管理」。

发布页只展示执行相关内容：

- 当前发布任务。
- 历史执行记录。
- 流程步骤。
- 服务列表。
- 发布日志。

发布参数统一放到「配置管理」：

- Jenkins 地址、用户、Token。
- 提交数阈值。

## 5. 验证场景

已覆盖后端核心测试：

- 相同 `id` 集合生成相同 `task_key`。
- `id` 缺失时使用 `relNo`。
- `id` 和 `relNo` 都缺失时使用规范化内容指纹。
- 服务快照去重并映射 Jenkins Job 名称。
- SQLite schema 初始化时包含发布任务表和执行记录表。

已执行验证：

- `cargo test --lib`
- `npm run build`
